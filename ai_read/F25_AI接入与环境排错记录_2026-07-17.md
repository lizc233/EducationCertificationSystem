## 3. 第一个错误：RabbitMQ 连接超时

### 3.1 现象

启动时出现：

- `AmqpConnectException`
- `Connection timed out: getsockopt`
- `Failed to check/redeclare auto-delete queue(s)`

### 3.2 初步判断

这类错误通常有三种可能：

1. RabbitMQ 容器没启动
2. 应用配置的主机名/端口不对
3. 应用所在环境和 RabbitMQ 不在同一个可通信网络里

### 3.3 后续现象变化

后面错误从超时变成了：

- `UnknownHostException: rabbitmq`

这个变化很关键。  
`连接超时` 和 `主机名无法解析` 不是一回事。

- 超时：说明“可能能找到地址，但连不上”
- `UnknownHostException`：说明“连地址都解析不出来”

### 3.4 真实原因

应用有两种运行方式：

1. 在 Docker 容器里运行
2. 在 IDEA/本机直接运行

这两种运行方式的 RabbitMQ 地址不能混用：

1. Docker 内运行时，RabbitMQ 可以写成容器服务名，例如 `rabbitmq`
2. IDEA 本机运行时，应该写 `localhost`

也就是说：

- `application-dev.yml` 应该偏向本机运行
- `application-docker.yml` 应该偏向容器内运行

### 3.5 修复思路

按运行环境拆配置：

1. `dev` 配置里 RabbitMQ 用 `localhost`
2. `docker` 配置里 RabbitMQ 用 `rabbitmq`

### 3.6 结论

RabbitMQ 本身不是核心问题，问题是“运行环境”和“主机名”混用了。

## 4. 第二个错误：Qdrant 容器明明启动了，但误以为没有向量库

### 4.1 现象

用户看到 Docker 里有：

- `education-certification-qdrant`

但一开始没有把它和“向量数据库”对应起来。

### 4.2 真实情况

它就是向量数据库容器，只是名字不是直接叫 `vector-db`。

需要区分两个概念：

1. Compose 服务名：`qdrant`
2. 容器名：`education-certification-qdrant`

访问地址也分两种：

1. IDEA 本机运行应用：`http://localhost:6333`
2. Docker 容器内运行应用：`http://qdrant:6333`

### 4.3 结论

Qdrant 没丢，问题只是名称理解和运行环境地址映射没有讲清楚。

## 5. 第三个错误：`ObjectMapper` Bean 缺失

### 5.1 现象

启动时报错：

- `QdrantVectorStoreService required a bean of type 'com.fasterxml.jackson.databind.ObjectMapper' that could not be found`

### 5.2 真实原因

AI 模块里新加的这些服务依赖 `ObjectMapper`：

1. `QdrantVectorStoreService`
2. `DeepSeekClientService`
3. `AiAssistantServiceImpl`
4. `AiKnowledgeIndexServiceImpl`

但当前项目里没有显式提供一个 `ObjectMapper` Bean。

### 5.3 修复动作

新增配置类：

- `src/main/java/com/educationcertificationsystem/config/JacksonConfig.java`

提供：

- `@Bean ObjectMapper`

### 5.4 结论

这是标准的 Spring Bean 装配问题，不是业务逻辑问题。

## 6. 第四个问题：DeepSeek Key 放哪里

### 6.1 问题本质

不是“能不能写 key”，而是“不同配置文件应该放什么”。

### 6.2 当前约定

1. `application.yml`：公共配置，保留占位符
2. `application-dev.example.yml`：示例配置，保留假值
3. `application-dev.yml`：本机开发配置，可读取环境变量，也可手工写真实值
4. `application-docker.yml`：容器运行配置，优先读环境变量

### 6.3 当前代码中的位置

DeepSeek 配置键：

- `app.ai.deepseek.base-url`
- `app.ai.deepseek.api-key`
- `app.ai.deepseek.model`

实际调用链：

1. `AiAssistantServiceImpl`
2. `DeepSeekClientService.generate(...)`
3. 请求 DeepSeek `/chat/completions`

### 6.4 排错过程中的一个配置坑

中间发现过一次 `application-dev.yml` 存在两个 `app:` 段。  
后面的 `app:` 会覆盖前面的 `app:`，导致前面写进去的 DeepSeek 配置可能根本不生效。

这个问题已经修正。

### 6.5 结论

Key 的本质不是“写不写”，而是“写在哪个 profile 下、有没有被覆盖”。

## 7. 第五个问题：Flyway 开着，但库里没有 `flyway_schema_history`

### 7.1 现象

观察到：

1. 数据库里已经有业务表
2. 已经有 `ai_knowledge_document`、`ai_knowledge_chunk`
3. 但是没有 `flyway_schema_history`

### 7.2 初步判断

这说明至少有一个结论成立：

- 当前数据库里的表，不完全是由当前这套 Flyway 历史记录机制管理出来的

### 7.3 为什么不能直接断定“Flyway 没有配置”

因为项目里确实已经有：

1. `spring.flyway.enabled=true`
2. `db/migration`
3. `V1.0__init_tables.sql`
4. `V1.1__f25_ai_rag_support.sql`

所以“代码里完全没配 Flyway”这个说法不成立。

### 7.4 进一步排查后发现的事实

启动日志里从头到尾没有任何 `Flyway` 相关信息。  
如果 Flyway 真装配成功，通常至少应该看到：

1. `Flyway Community Edition ...`
2. `Schema history table ...`
3. `Migrating schema ...`

但实际一行都没有。

这说明问题不是“Flyway 执行失败”，而是：

**Flyway 根本没有参与这次 Spring Boot 启动。**

### 7.5 为什么库里已经有表

仓库根目录本来就有：

- `数据库建表.sql`

所以当前数据库里很多表，大概率是之前直接执行整库脚本建出来的，不是 Flyway 管出来的。

### 7.6 这次定位到的真实根因

前面先补了：

1. `flyway-core`
2. `flyway-mysql`

但进一步核对后，当前 Spring Boot 4 启动链路更稳妥的接法应该是：

1. `spring-boot-starter-flyway`
2. `flyway-mysql`

也就是说，问题不是用户没有刷新 Maven，而是依赖接法本身不够完整，导致：

- IDEA 里能看到 Flyway 依赖
- 但 Spring Boot 自动配置没有真正把 Flyway 拉起来

### 7.7 当前修复动作

`pom.xml` 已调整为：

1. `org.springframework.boot:spring-boot-starter-flyway`
2. `org.flywaydb:flyway-mysql`

### 7.8 当前状态说明

截至 2026-07-17 本次记录结束时：

1. 依赖已改
2. 根因已经大幅收敛
3. 但还需要重新启动后，用启动日志确认 `Flyway` 已实际装配并执行

也就是说，这一项属于：

- 修复方案已落地
- 最终运行结果仍需二次验证

## 8. 这次排错的标准流程

以后再遇到类似问题，可以按这个顺序查：

### 8.1 先分环境

先确认应用到底是在哪跑：

1. IDEA 本机
2. Docker 容器

这个判断会直接影响：

1. RabbitMQ 地址
2. Qdrant 地址
3. MySQL 地址
4. profile 选择

### 8.2 再看错误类型

不要把不同错误混成一个问题。

例如：

1. `Connection timed out` 是连不上
2. `UnknownHostException` 是主机名都解析不了
3. `Bean not found` 是 Spring 装配问题
4. “启动成功但没执行迁移”是自动配置/依赖链问题

### 8.3 再看日志有没有关键组件痕迹

例如你怀疑某个框架是否参与启动，不要靠猜，直接看日志里有没有它的名字：

1. RabbitMQ
2. Flyway
3. MyBatis
4. Tomcat

如果一行都没有，优先怀疑“没有装配成功”，而不是先怀疑业务代码。

### 8.4 再核对 profile 和配置文件覆盖关系

尤其是 Spring Boot 项目里，要注意：

1. `application.yml`
2. `application-dev.yml`
3. `application-docker.yml`

以及是否存在重复键覆盖，例如两个 `app:` 段。

### 8.5 再核对依赖是否只是“看得见”，还是“真生效”

IDEA 的 External Libraries 里能看到依赖，不等于 Spring Boot 自动配置一定生效。

还要结合：

1. 启动日志
2. 实际 Bean 行为
3. 目标表/目标连接/目标接口是否真的出现

## 9. 这次排错得到的经验

### 9.1 配置分层必须清楚

`application`、`application-dev`、`application-docker` 不是一回事。  
公共默认值、示例值、开发机真实值、容器环境值必须分开。

### 9.2 看到表存在，不代表迁移机制存在

数据库里“有表”只能证明“某处建过表”，不能证明“Flyway 正在管它”。

### 9.3 先判断框架有没有参与启动

如果启动日志里没有 `Flyway`，那就先查自动配置和依赖，不要先盯着 SQL 文件。

### 9.4 先分清“网络问题”和“Bean 问题”

RabbitMQ 的超时、Qdrant 的地址、`ObjectMapper` 的缺失、Flyway 的不装配，属于四类不同问题，不能混着修。
